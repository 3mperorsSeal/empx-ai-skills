# EmpX Router Skill

EmpX is a decentralized, on-chain DEX Aggregator liquidity router optimized specifically for AI agents and autonomous trading systems. This infrastructure allows AI systems to discover, simulate, and execute deterministic token swaps across multiple EVM-compatible blockchains.

## Core Principles for AI Agents
1. **Never guess routes:** Always simulate first.
2. **Deterministic Execution:** The input variables (`adapters`, `path`, `amounts`) must be perfectly synchronized with the simulation results.
3. **Safe Reverts:** Enforce an `amountOut` minimum to ensure the swap reverts cleanly if the market moves unfavorably.

---

## 🏗️ Capability Definitions (`skills/`)

The `skills/` directory contains machine-readable JSON files defining the three core capabilities. These are structured to be ingested by AutoGPT, Claude, or custom LLM planners.

1. **`empx-best-path`** (Simulation, Safe/View only)
   - Discovers the optimal multi-hop route for a swap.
   - Takes `chain_id`, `token_in`, `token_out`, `amount_in` (in wei), and `max_steps`.
   - Returns a `FormattedOffer` via the `Trade` schema.

2. **`empx-gas-aware-route`** (Simulation, Safe/View only)
   - Discovers the optimal route while dynamically pricing the gas cost into the calculation.
   - Takes the same inputs as best path, plus the current `gas_price` (in wei).
   - Essential for ensuring small swaps are actually profitable after gas fees.

3. **`empx-swap`** (Execution, Mutates State)
   - Executes the token swap on-chain.
   - Must be passed the exact `adapters` array, `path` array, and expected `amountOut` derived from either of the simulation skills.

**Critical JSON Fields in `chains/<network>.json`:**
* `router`: The main contract address you must call for simulations and executions.
* `wrapped_native`: The WETH/WPLS equivalent for the network.
* `adapters`: An array of DEX adapter contracts. The simulation engine will automatically determine the best subset of these to use for routing.
* `native_swap_from_func` & `native_swap_to_func`: If an agent is swapping *from* or *to* the native gas token (e.g. raw ETH instead of WETH), the agent **must** map the execution call to this specific function signature instead of the generic `swapNoSplit` function.

## ⚙️ Execution Workflow

When constructing an autonomous workflow, AI tools should adhere to the following sequence:

### 1. Read Configuration
Load the target chain's JSON file (`chains/<network>.json`). Extract the `router` address, `wrapped_native` token address (if needed), and the list of `adapters`.

### 2. Simulate via RPC
Use the `empx_gas_aware_route` (or `empx_best_path`) capability. 
* Build an `eth_call` targeting the `router` address. 
* Invoke the `findBestPathWithGas` view function on the contract. 
* This call is free and will not mutate state on the blockchain.
* The response will be exactly conforming to the `schemas/Trade.schema.json` structure (e.g., `amountOut`, `path`, `adapters`).

### 3. Evaluate Profitability
Compare the simulated `amountOut` against the inputs and the `gasEstimate`. If the trade is profitable according to the agent's strategy, proceed to execution. Calculate a minimum acceptable output (slippage buffer), usually around `amountOut * 0.995` (0.5% slippage).

### 4. Execute Swap Transaction
Use the `empx-swap` capability.
* Construct an `eth_sendRawTransaction`.
* Pass the exact `path` array (the tokens to route through) and `adapters` array (the DEX protocols used at each hop) generated by the simulation.
* Pass the calculated slippage floor as `amountOut`.
* Determine which wrapper function to call based on the chain config's `native_swap_to_func`/`native_swap_from_func` attributes if dealing in the native gas asset.

## ⚠️ Critical Agent Parameters & Constraints

* **Units:** All numeric amounts (`amountIn`, `amountOut`, `gasPrice`, `gasEstimate`, `fee`) are denominated in **wei**. Agents must understand token decimals (typically 18, but USDC/USDT is often 6) when calculating these integer strings.
* **Routing Limits:** The `maxSteps` parameter controls routing hops. Valid values are `1` through `4`.
* **API vs RPC:** While `openapi.yaml` defines standard REST endpoints (e.g., `/simulate/path`), the underlying infrastructure expects direct RPC interaction (`eth_call` for simulation, `eth_sendRawTransaction` for execution) against the router contract if the REST wrapper service is not actively deployed in the agent's environment.
* **Schemas:** Validate all outputs from simulation and inputs for execution against `schemas/Trade.schema.json` and `schemas/Query.schema.json` to ensure type safety before interacting with the blockchain.
